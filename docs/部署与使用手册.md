# 部署与使用手册

## 1. 环境要求
- MySQL 8.x（已启动）
- Python 3.10+
- JDK 17+
- Maven 3.9+
- Node.js 18+

数据库默认参数：
- Host：`localhost`
- Port：`3306`
- User：`root`
- Password：`你的MySQL密码`
- DB：`video_analysis`

建议先设置环境变量（避免每次输入）：

```powershell
$env:MYSQL_USER="root"
$env:MYSQL_PASSWORD="你的MySQL密码"
```

## 2. 一键部署（推荐）
在项目根目录双击：
- `OneClick_Full_Auto_Launch.bat`

它会自动执行：
1. Python 依赖安装
2. 数据库初始化（无需 MySQL CLI）
3. 启动后端与前端
4. 运行自测并输出报告
5. 打开浏览器

说明：

- 如果未设置 `MYSQL_PASSWORD`，脚本会在运行时提示输入 MySQL 密码。
- 首页已使用聚合接口 `/video/dashboard` 降低请求次数，属于无感优化。

## 8. 轻量性能优化（可选）

## 8.1 导入速度优化（不改功能）

后端支持配置行为日志写入强度：

- 配置项：`CRAWLER_BEHAVIOR_SCALE`
- 默认值：`0.35`（已优化为更快的默认值）
- 示例：设置为 `0.2` 可进一步减少行为日志写入量（视频主数据不受影响）

后端还支持单次导入行为日志上限：

- 配置项：`CRAWLER_BEHAVIOR_MAX_ROWS`
- 默认值：`300000`
- 作用：限制单次导入生成的 `user_behavior` 总量，避免超大导入导致长时间卡住。

示例（临时会话）：

```powershell
$env:CRAWLER_BEHAVIOR_SCALE="0.2"
$env:CRAWLER_BEHAVIOR_MAX_ROWS="200000"
```

说明：

- 以上配置不会减少视频主数据（`video` 表）导入条数。
- 主要影响行为日志规模，从而提升“导入”和“清理”成功率与响应速度。

## 8.4 导入/清理慢或难成功时的排查

1. 确认后端已重启并加载新配置：
- 执行 `OneClick_Full_Auto_Launch.bat`，或重启后端窗口。

2. 确认前端超时配置为无限等待（长任务场景）：
- `start_frontend.ps1` 已内置 `VITE_API_TIMEOUT_MS=0`。

3. 参考实测（本机，大文件 `BiliBili_data.csv`）：
- 优化前：导入约 134 秒，清理约 206 秒。
- 优化后：导入约 16 秒，清理约 11 秒。

## 7. 常见问题
1. 登录接口 404：
- 原因：后端仍是旧进程。
- 处理：关闭旧后端后重新 `mvn spring-boot:run`。

2. 前端能打开但数据接口 401：
- 原因：未登录或登录已过期。
- 处理：返回首页重新登录。

3. 初始化时报列不存在：
- 处理：使用最新 `scripts/init_db_python.py`，它会自动补齐旧库字段。


## 9. 用户画像异常排查（新增）

若出现“用户画像几乎全部同一标签”（例如全是高互动讨论型），请按以下步骤处理：

1. 重启后端，确保加载最新画像规则。
2. 在“数据管理”执行清除后重新导入（旧行为日志不会自动重算）。
3. 导入完成后到“用户分析”查看标签分布，正常应出现多标签而非单一标签。

## 10. 用户画像全同标签时的修复说明（2026-03-10）

现象：用户分析页面出现“几乎全部用户同一画像标签”。

原因：历史行为日志分布过于集中，固定阈值会把用户批量归到同一标签。

处理：
1. 已在后端升级画像规则为“固定阈值 + 分位数阈值”。
2. 重启后端生效。

验证：
1. 登录系统后打开“用户分析”。
2. 观察“用户画像标签分布”是否出现多标签（如 steady_viewer / high_frequency_active / high_interaction_discuss）。
