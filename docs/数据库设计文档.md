# 数据库设计文档

更新时间：2026-03-09

## 1. 设计目标

数据库围绕“多租户视频数据分析”设计，目标如下：

1. 按账号隔离数据（`tenant_user_id`）。
2. 支持视频主数据、行为数据、统计数据与导入任务数据。
3. 支持导入质量追踪与拒绝记录回溯。
4. 保持对旧库的兼容升级能力。
5. 在不改业务功能前提下支持轻量性能优化。

## 2. 数据库与字符集

- 数据库名：`video_analysis`
- 字符集：`utf8mb4`
- 排序规则：`utf8mb4_unicode_ci`

## 3. 当前运行主链路表（已启用）

以下表由 `database/init.sql` 与后端启动时 `SchemaUpgradeRunner` 共同保障：

1. `app_user`：账号表
2. `app_session`：会话表
3. `video`：视频主表
4. ``user``：作者/用户表
5. `comment`：评论表
6. `user_behavior`：行为日志表
7. `video_statistics`：视频日统计表
8. `user_interest_result`：用户兴趣画像结果表
9. `import_job`：导入任务表
10. `import_reject_record`：低质量/拒绝记录表

## 4. 核心表结构（摘要）

## 4.1 `app_user`

- 主键：`id`
- 关键字段：`username`（唯一）、`password_hash`、`password_salt`
- 说明：支持注册登录、密码升级策略

## 4.2 `app_session`

- 主键：`token`
- 关键字段：`user_id`、`expires_at`、`is_revoked`
- 说明：无状态 token + 服务端会话有效性控制

## 4.3 `video`

- 主键：`id`
- 租户字段：`tenant_user_id`
- 关键字段：
  - 基础：`title`、`author`、`category`、`publish_time`
  - 平台：`source_platform`、`source_url`、`platform_video_id`
  - 指标：`play_count`、`like_count`、`comment_count`、`share_count`、`favorite_count`
  - 入库：`import_type`、`source_file`、`import_time`、`import_job_id`
  - 质量：`data_quality_score`、`quality_level`、`ai_confidence`
  - 扩展：`tags_json`、`extra_json`
- 唯一索引：`uk_video_tenant_dedupe(tenant_user_id, dedupe_key)`

## 4.4 ``user``

- 主键：`user_id`
- 租户字段：`tenant_user_id`
- 关键字段：`user_name`、`fans`、`follow`、`level`

## 4.5 `user_behavior`

- 主键：`id`
- 租户字段：`tenant_user_id`
- 关键字段：`user_id`、`video_id`、`action`、`time`
- 说明：用于互动效率、用户画像等分析

## 4.6 `import_job`

- 主键：`id`
- 租户字段：`tenant_user_id`
- 关键字段：`import_type`、`source_platform`、`source_file`、`source_count`、`success_count`、`status`
- 说明：记录每次导入任务执行情况

## 4.7 `import_reject_record`

- 主键：`id`
- 租户字段：`tenant_user_id`
- 关键字段：`import_job_id`、`reject_reason`、`suggest_fix`、`raw_excerpt`、`quality_score`、`ai_confidence`
- 说明：用于导入失败反馈和可修复建议

## 6. 迁移机制说明

本项目采用双层迁移机制：

1. 静态 SQL：`database/init.sql`
2. 启动补偿迁移：`SchemaUpgradeRunner`

设计原因：

- 兼容旧版本数据库字段缺失场景
- 保证一键启动时数据库可用
- 允许“主链路稳定 + 性能脚本按需执行”的渐进优化策略

## 10. 2026-03-09 性能优化补充（功能不变）

本次新增/调整数据库对象如下：

1. 新增表 `tenant_overview_cache`
   - 主键：`(tenant_user_id, source_platform)`
   - 字段：`video_count/user_count/comment_count/behavior_count/total_play_count/source_platform_count/refreshed_at`
   - 用途：缓存首页总览统计，避免每次请求扫描大表和多次联表聚合。

2. 新增索引 `idx_behavior_tenant_user_video`
   - 表：`user_behavior`
   - 列：`(tenant_user_id, user_id, video_id)`
   - 用途：优化用户画像模块中按租户+用户聚合并回表 video 的查询路径。

3. 迁移与兼容
   - `database/init.sql` 已包含上述结构；
   - `SchemaUpgradeRunner` 启动时会自动补齐表/索引并预热总览缓存；
   - `database/upgrade_performance_light.sql` 已追加同等升级项，支持已有库按需手工升级。

## 11. 2026-03-09 清理功能一致性补充

本次未新增数据库字段/索引，但调整了清理事务的执行顺序：

1. 先删除 `import_reject_record`，再删除 `import_job`。
2. 该顺序在存在潜在外键约束或触发器的环境中更稳健，可降低清理失败概率。
3. 清理后会刷新 `tenant_overview_cache`，确保首页总览不出现“表面清理”。

## 12. 2026-03-10 导入/清理时长说明（运行特性）

1. 在全量数据导入策略下，`user_behavior` 会生成较大规模记录，因此导入与清理均可能达到分钟级。
2. 本次未新增表结构，但已通过前端与脚本配置确保长事务请求不被 10 秒超时中断。
3. 建议运维侧保持后端窗口常驻，不在导入/清理过程中关闭后端进程。

## 13. 2026-03-10 性能与稳定性补充（数据库相关）

本次未新增业务表字段，主要是导入/清理执行策略优化：

1. 写入规模控制（配置项）
- `crawler.import.behavior-scale` 默认值调整为 `0.35`。
- 新增 `crawler.import.behavior-max-rows-per-import`（默认 `300000`）。
- 作用：控制 `user_behavior` 生成规模，避免超大导入造成长时间阻塞。

2. 清理策略优化
- `user_behavior` 由单次大删除改为分批删除（`DELETE ... LIMIT 50000` 循环）。
- 减少单条 SQL 长耗时与锁持有时间，提高清理成功率。

3. 缓存一致性策略
- 导入/清理后不再立即全量重算 `tenant_overview_cache`，改为直接失效该租户缓存。
- 首次读请求按实时数据回填缓存，降低写路径压力。

4. 一致性说明
- `video` 主数据仍按原规则全量入库。
- 上述策略只影响导入速度与行为日志规模，不改变对外接口结构。
